1. System Architecture Overview
The KodetoCareer platform follows a multi-tenant, microservice-inspired monolith architecture for the MVP
phase, transitioning to true microservices at scale. This approach balances development speed with long-term
scalability.
Architecture Pattern
 Backend: Modular Monolith (Node.js / Express.js) with clear domain boundaries
 
 Frontend Web: React.js (Single Page Application)
 
 Mobile: Flutter (iOS + Android, single codebase)
 
 Database: PostgreSQL (primary), Redis (caching & sessions)
 
 Storage: AWS S3 + CloudFront CDN
 
 Video Processing: AWS MediaConvert
 
 Authentication: JWT (access tokens) + Refresh tokens
 
 Real-time: Socket.io for live class chat and notifications
 
 Search: PostgreSQL full-text search (MVP), Elasticsearch at scale

5. Authentication & Security
5.1 Token Strategy
Access Token:
  - JWT signed with RS256 (asymmetric)
  - Payload: { userId, role, collegeId, batchIds[], iat, exp }
  - Expiry: 15 minutes
  - Stored in: memory (web), flutter_secure_storage (mobile)
Refresh Token:
  - Opaque random string (64 bytes)
  - Stored in: httpOnly cookie (web), secure storage (mobile)
  - Expiry: 30 days
  - Rotated on each use (refresh token rotation)
  - Stored in Redis with user ID mapping for instant revocation
5.2 Security Requirements
TR-SEC-1 — All API endpoints must use HTTPS. HTTP requests must redirect to HTTPS.
TR-SEC-2 — Rate limiting: 100 requests/minute per IP for public endpoints, 1000/minute for authenticated users.
TR-SEC-3 — File uploads must be virus-scanned using ClamAV or AWS GuardDuty before being served.
TR-SEC-4 — All S3 buckets must be private. Content served only through signed CloudFront URLs with 1-hour
expiry.
TR-SEC-5 — SQL injection prevention: Prisma ORM with parameterised queries only. No raw SQL string
concatenation.
TR-SEC-6 — Sensitive fields in the database (not passwords) that should never appear in API responses:
internal notes, payment details.
TR-SEC-7 — Passwords must be hashed with bcrypt (minimum 12 rounds).
TR-SEC-8 — API must implement CORS with a strict whitelist of allowed origins.
TR-SEC-9 — Admin panel must enforce IP allowlisting in production (configurable).
TR-SEC-10 — All admin actions (create, update, delete) must be logged in an audit_logs table.

6.2 Video Processing Pipeline
1. Trainer uploads MP4 → Multipart upload to S3 raw bucket
2. S3 event triggers Lambda function
3. Lambda submits MediaConvert job:
   - Output: HLS (480p, 720p, 1080p) + MP4 (720p for download-optional)
   - Thumbnail extraction at 10-second mark
4. MediaConvert completes → processed files written to S3 processed bucket
5. Lambda updates database: lesson.videoStatus = 'READY', lesson.videoUrl = CDN URL
6. Push notification sent to admin/trainer confirming video is live

9. Real-Time Requirements
Socket.io shall be used for:
 Live class session chat (students ask questions during recorded/live classes)
 
 Real-time attendance count updates (trainer sees count update as they mark)
 
 Notification delivery (push to open browser tabs)
 
 Live quiz countdown (if exam mode is enabled)
 
Socket.io namespace strategy:
/notifications — general user notifications
/class/{sessionId} — class-specific chat
/quiz/{quizId}/{attemptId} — quiz session state
